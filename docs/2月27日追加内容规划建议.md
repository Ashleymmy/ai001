### 33. 修复视频时间线预览画面抽搐问题

| 维度 | 状态 |
|------|------|
| **完成度** | ★☆☆☆☆（需修复） |
| **现状** | 问题已定位，尚未修复 |

**问题分析（PreviewPlayer.tsx）：**

经代码分析，发现以下导致画面抽搐/闪烁的具体原因：

1. **每帧 setState 调用**（第64-107行）：图片镜头播放时使用 `requestAnimationFrame` 循环，每帧都调用 `setElapsedInShot()`，导致 ~60次/秒的 React 状态更新和重渲染
2. **多个 useEffect 重叠触发**（第59、109、119行）：切换镜头时 `current?.id` 和 `current?.video_url` 同时变化，触发多个 effect 在同一渲染周期执行，导致短暂视觉跳闪
3. **Video 元素 key 强制重建**（第178行）：`key={current.id_${current.video_url}}` 导致每次切镜头都完全销毁并重建 `<video>` DOM 元素，造成可见的空白闪烁
4. **无预加载策略**：下一个镜头的视频需要从零开始加载缓冲，切换时有明显延迟
5. **onTimeUpdate 粒度低**（第183-187行）：浏览器 `onTimeUpdate` 约250ms触发一次，进度条呈阶梯式跳动而非平滑

**修复方案建议：**

```
修复优先级排序：

[P0] 消除每帧 setState
   → 使用 useRef 存储 elapsedInShot，仅在进度条/时间显示更新时
     以较低频率（如 10fps）同步到 state

[P0] 消除 video 元素重建闪烁
   → 使用双缓冲策略：维护两个 <video> 元素，一个播放当前镜头，
     另一个在后台预加载下一个镜头，切换时直接交换可见性

[P1] 合并 useEffect
   → 将多个监听相同依赖的 effect 合并为单个，
     避免同一渲染周期内多次触发

[P1] 添加视频预加载
   → 在当前镜头播放到 70% 时，开始预加载下一个镜头的视频

[P2] 平滑进度条
   → 使用 CSS transition 或 requestAnimationFrame 插值，
     让进度条在 onTimeUpdate 事件之间平滑过渡
```

**优先级：高（直接影响用户体验的明显 Bug）**

**现状分析：**

当前协作体系的局限性：
- `collab_service.py` 仅提供 JWT 认证 + workspace CRUD + 角色管理（owner/editor/viewer）
- 数据隔离方式：series 表有 `workspace_id` 列，通过 WHERE 条件过滤

```
- element_updated（素材变更通知）
- shot_updated（镜头变更广播，用于只读预览同步）
```

**5. 性能考量**
```
- SQLite 单写者瓶颈：当前 WAL 模式下并发写入是串行化的
  → 短期可接受（Electron 桌面端，2-5 人协作）
  → 长期如需 10+ 人并发，需迁移到 PostgreSQL

- 素材文件存储：当前本地文件系统
  → 协作场景需共享存储：局域网共享目录 或 S3 兼容对象存储
  → 建议增加存储后端抽象层（local / s3 / smb）

- 前端状态同步：
  → studioStore 增加 WebSocket 监听，接收远程变更事件
  → 使用乐观更新 + 冲突回退策略
```

**优先级：高（协作是核心需求，但工程量大，建议分 3 个子阶段）**

子阶段：
- **36-A**：Episode 分配 + 写锁 + 提交审核（最小可行协作）
- **36-B**：WebSocket 实时通知 + 在线状态
- **36-C**：独立操作 Dashboard + 合并审核台

### 38. 超长篇小说分节制作的风格迁移及统一问题

| 维度 | 状态 |
|------|------|
| **完成度** | ☆☆☆☆☆（0%） |
| **现状** | 完全不支持 |

**现状问题：**

当前 Studio 的数据层是**严格两级结构**：`series → episodes`（通过 `act_number` 排序）。

关键缺失：
1. **无"卷/章"概念**：数据库中搜索 "volume"/"chapter"/"卷"/"章" 返回零结果
2. **无增量添加 Episode 的端点**：episodes 只在 `create_series` 时从脚本批量拆分生成，没有 `POST /api/studio/series/{id}/episodes` 端点
3. **无跨章节风格迁移**：每个 episode 的 `creative_brief` 和生成结果相互独立
4. **脚本上传一次性**：series 的 `source_script` 是一次性写入，无追加机制

对于一部 50 万字的长篇小说，当前流程完全不可行：
- 一次性粘贴 50 万字 → LLM 上下文溢出
- 无法按章节逐步推进制作
- 第 1 章和第 30 章的画风可能完全不一致

**规划建议：**

#### 数据层改造：三级结构 `series → volumes → episodes`

```sql
-- 新增 volumes 表
CREATE TABLE IF NOT EXISTS volumes (
    id              TEXT PRIMARY KEY,
    series_id       TEXT NOT NULL REFERENCES series(id) ON DELETE CASCADE,
    volume_number   INTEGER NOT NULL,
    title           TEXT DEFAULT '',
    summary         TEXT DEFAULT '',
    source_text     TEXT DEFAULT '',      -- 本卷原文
    style_anchor    TEXT DEFAULT '{}',    -- 本卷风格锚点（从首集提取）
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL
);

-- episodes 表增加 volume_id 外键
ALTER TABLE episodes ADD COLUMN volume_id TEXT DEFAULT '' REFERENCES volumes(id);
```

#### 风格迁移机制

```
问题：第 1 卷制作完后，第 2 卷要延续同样的视觉风格

解决方案：Style Anchor Chain（风格锚点链）

┌─────────┐    style_anchor    ┌─────────┐    style_anchor    ┌─────────┐
│  第1卷    │ ───────────────→  │  第2卷    │ ───────────────→  │  第3卷    │
│          │                   │          │                   │          │
│ 制作完成后 │   传递以下锚点：    │ 新卷制作时 │                   │          │
│ 提取锚点  │   • visual_style  │ 自动继承  │                   │          │
│          │   • 共享角色素材图   │ 上卷锚点  │                   │          │
│          │   • 色彩/光影基调   │          │                   │          │
│          │   • 画风参考帧     │          │                   │          │
└─────────┘                   └─────────┘                   └─────────┘
```

具体实现：

**1. 风格锚点数据结构**
```json
{
  "visual_style": "国风叙事动画插画风格",
  "color_palette": ["#2C1810", "#8B4513", "#DEB887", "#F5DEB3"],
  "reference_frames": ["url_of_best_frame_vol1_1", "url_of_best_frame_vol1_2"],
  "character_reference_map": {
    "主角": "element_image_url",
    "女主": "element_image_url"
  },
  "negative_prompt": "...",
  "generation_params": {
    "default_width": 1280,
    "default_height": 720,
    "style_strength": 0.8
  }
}
```

**2. 新卷创建流程**
```
用户操作：在 series 中点击"添加新卷" → 上传/粘贴新章节文本
系统自动：
  1. 创建 volume 记录
  2. 继承上一卷的 style_anchor
  3. 继承 series 级的 shared_elements（角色/场景）
  4. 对新文本执行 script_split → 生成 episodes
  5. 在每个 episode 的规划提示词中注入 style_anchor 作为风格约束
  6. 帧生成时，将上一卷的参考帧作为 reference images 传入
```

**3. 前端改造**
```
Series 侧边栏改造：

当前：
  系列名
  ├── 第1集
  ├── 第2集
  └── 第3集

改造后：
  系列名
  ├── 📖 第1卷：少年篇
  │   ├── 第1集：出发
  │   ├── 第2集：相遇
  │   └── 第3集：初战
  ├── 📖 第2卷：成长篇
  │   ├── 第4集：修炼
  │   └── 第5集：突破
  └── [+ 添加新卷]  ← 新增入口
```

**4. 新增 API 端点**
```
POST   /api/studio/series/{series_id}/volumes                 — 创建新卷
GET    /api/studio/series/{series_id}/volumes                 — 列出所有卷
PUT    /api/studio/volumes/{volume_id}                        — 更新卷信息
DELETE /api/studio/volumes/{volume_id}                        — 删除卷
POST   /api/studio/volumes/{volume_id}/episodes               — 在卷内追加集
POST   /api/studio/volumes/{volume_id}/extract-style-anchor   — 提取风格锚点
POST   /api/studio/series/{series_id}/migrate-style           — 跨卷风格迁移
```

**5. 风格一致性保障策略**
```
• 帧生成时：style_anchor 的 reference_frames 作为 ControlNet / IP-Adapter 参考
• 视频生成时：style 描述注入 video_prompt
• 角色一致性：共享素材图跨卷复用，不重新生成
• 色彩一致性：color_palette 注入提示词（"色调以暖棕色为主"）
• 自动检测：对比新旧卷首帧的色彩分布，超过阈值则警告
```

**优先级：高（解决超长篇内容制作的根本性问题，但工程量最大）**
