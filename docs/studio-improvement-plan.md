# Studio 长篇制作工作台 — 测试后改进详细规划

> 基于 `docs/测试后修改建议.txt` 中 24 条建议，结合当前代码现状分析，制定分阶段实施方案。

---

## 现状摘要

| 层 | 已完成 | 存在但未接入 UI | 完全缺失 |
|----|--------|----------------|----------|
| **后端存储** | 5 张表 CRUD、快照导出 | reorder_shots、per-series settings 覆盖、image_history | 版本历史/Diff |
| **后端服务** | 创建系列、规划/增强、素材生成、批量生成 | — | 真实文件导出/打包、进度上报（SSE）、提示词哨兵 |
| **前端 Store** | 全部 action 已接 API | — | 进度粒度不够（仅 3 个 boolean） |
| **前端 UI** | 系列/集/镜头浏览、镜头详情编辑、设置面板（LLM/Image/Video） | 共享元素 CRUD 按钮、集删除、镜头删除 | 素材库、时间线、导出、预览、动画交互、自适应布局 |

---

## 阶段划分

```
Phase 1（P1）: 核心体验补全 — 让现有功能闭环可用
Phase 2（P2）: 素材与编辑 — 素材管理 + 编辑能力
Phase 3（P3）: 预览与导出 — 时间线 + 导出打包
Phase 4（P4）: 交互与体验 — 动画、布局、容错
Phase 5（P5）: 智能增强 — 提示词哨兵、Agent 协同、高级功能
```

### 2.2 素材编辑功能（建议 #5）

**现状**：素材（参考图、帧、视频）生成后无编辑能力。

**改动方案**：为图像类素材增加基础编辑入口。

| 功能 | 描述 | 实现方式 |
|------|------|----------|
| 重做/重新生成 | 对不满意的参考图/帧一键重做 | 调用已有 generate API，保留历史 |
| 图片历史 | 展示该元素/镜头的所有历史图片，可切换选用 | `image_history` 字段已存在 |
| 提示词微调后重做 | 编辑 prompt 后重新生成，对比前后效果 | 前端增加编辑弹窗 |
| 局部调整（inpaint） | 选区 + 补充描述进行局部重绘 | 需后端新增 inpaint 接口（依赖 ImageService 是否支持） |

**后端改动**：
- `studio_service.py` — `generate_element_image()` 和 `generate_shot_frame()` 生成时将旧图存入 `image_history`
- `studio_service.py` — 新增 `inpaint_shot_frame(shot_id, mask_data, edit_prompt)` 方法（如 ImageService 支持）
- `main.py` — 新增 `POST /api/studio/shots/{id}/inpaint` 路由

### 3.2 复用导出打包功能（建议 #10）

**现状**：Agent 有完整 ExportService（ZIP 打包 + FFmpeg 视频拼接），Studio 仅返回 JSON 快照。

**改动方案**：为 Studio 适配 ExportService，或新建 StudioExportService。

**导出功能清单**：

| 功能 | 描述 |
|------|------|
| 单集素材导出 | 该集所有帧图片 + 视频 + 音频 + 元数据 → ZIP |
| 单集视频合并 | FFmpeg 拼接该集所有视频片段 → 单文件 |
| 全系列导出 | 所有集的素材 → 按集分目录的 ZIP |
| 全系列合并 | 所有集视频按顺序合并 → 单文件 |
| 导出元数据 | SRT 字幕文件（旁白/对白）+ 分镜表（JSON/CSV） |

**实现方式**：
- 新建 `backend/services/studio_export_service.py`
- 复用 `export_service.py` 中的 FFmpeg 调用逻辑（抽取为公用函数）
- Studio 数据结构 → ExportService 所需格式的适配器

### 4.5 容错与恢复（建议 #18 延伸）

**现状**：网络抖动或快速切换系列/集时，存在请求竞争与短暂空白风险；失败后缺少就地恢复入口。

| 功能 | 描述 |
|------|------|
| 读接口瞬断重试 | 列表/详情/历史加载遇到 `network_error` / `network_timeout` 自动重试一次 |
| 请求乱序保护 | 为系列/集/历史加载增加请求序号，忽略过期响应，避免旧数据覆盖新选择 |
| 就地恢复条 | 顶部显示网络异常提示条，提供「重试当前上下文」按钮 |
| 历史 Diff 容错 | 历史面板默认携带 snapshot，缺失时退化为摘要展示，不阻断回退 |
| 失败队列与重试历史 | 记录失败操作（可重试/可移除），并提供重试历史可视化面板，支持清理已恢复项 |

**实现方式**：
- Frontend Store：`withTransientRetry + requestSeq`（无新增依赖）
- Frontend UI：在 `StudioPage` 顶部插入可关闭的恢复提示条
- Frontend UI：新增「恢复中心」浮层，展示失败操作队列与重试历史（支持重试/清理）
- 后端接口保持兼容，不破坏原有调用

**预计工作量**：前端 ~120 行
