# 精细化视频制作提示词集成规划

> 生成日期：2026-02-27
> 参考文件：`docs/prompt_i18n.go`（drama-generator 项目提示词体系）
> 目标：将其专业级影视提示词理念深度融合到当前 Studio 模块

---

## 一、背景与目标

`prompt_i18n.go` 来自一个成熟的短剧制作系统（drama-generator），包含 **9 大提示词模块**，覆盖了从大纲规划到帧级生成的完整链路。其核心优势：

| 能力 | Go 文件 | 当前 Studio |
|------|---------|-------------|
| 分镜拆解（独立动作单元） | 完整的罗伯特·麦基理论 | 无独立分镜拆解，依赖 episode_planning 一步生成 |
| 景别标准（ELS/LS/MS/CU/ECU） | 6 级专业景别 | 5 种泛化类型（standard/quick/closeup/wide/montage） |
| 运镜词汇表 | 固定/推/拉/摇/跟/移 6 种 | visual_action 有类似词汇，但不注入规划提示词 |
| 情绪强度标记 | 5 级标记（↑↑↑ 到 ↓）逐镜头 | 仅 episode 级 emotional_tone |
| 首帧/关键帧/尾帧三帧分化 | 三套独立提示词，语义明确 | 仅 start_frame/end_frame，无 key_frame |
| 场景纯背景提取 | 专用提示词，明确"无人物" | element_extraction 混合提取角色/场景/道具 |
| 角色详细提取 | 含 role 分级、外貌/性格/背景 | 有但缺 personality/description 分离 |
| 道具独立提取 | 含 image_prompt 生成字段 | 混在 element_extraction 中，无独立 image_prompt |
| 风格/比例注入 | 全链路统一注入 | 风格注入有但比例仅通过尺寸参数传递 |
| 国际化 | 中英双语完整支持 | 仅中文 |

**目标**：不是简单移植 Go 代码，而是将其专业影视提示词设计理念与当前 Studio 的 Python/React 架构深度融合，提升生成质量。

### 3.2 改造 episode_planning 提示词（注入专业分镜理论）

**文件**: `backend/services/studio/prompts.py` — `episode_planning` 模块

**当前问题**：
- episode_planning 的 system_prompt 要求的 shot `type` 只有 5 种泛化类型
- 不要求 LLM 输出运镜、机位角度、情绪强度等专业字段
- 没有"独立动作单元"拆解原则

**改造方案**：

在 episode_planning 的 **system_prompt** 中注入 Go 文件的分镜拆解原则：

```
追加到现有 system_prompt：

【分镜拆解原则】（参考罗伯特·麦基镜头拆解理论）
1. **动作单元划分**：每个镜头对应一个完整且独立的动作
   - 一个动作 = 一个镜头（站起、走过去、说台词、做表情反应等）
   - 避免在单个镜头中合并多个独立动作
2. **景别选择须符合叙事节奏**：不要连续使用同一景别
3. **情绪标记**：每个镜头须标注当前情绪和情绪强度等级
```

在输出 JSON 的 shots 结构中 **新增字段**：

```json
{

  "type": "standard/quick/closeup/wide/montage",

  "camera_movement": "fixed/push/pull/pan/follow/tracking/orbit",
  "emotion": "紧张",
  "emotion_intensity": 2,
  "duration": 6.0,
  "description": "...",
  "prompt": "首帧提示词 with [SE_XXX]",

async def _build_enhanced_video_prompt(self, shot, episode_id):
    """增强版视频提示词（融合 Agent 的专业要素）"""
    raw = shot.get("video_prompt") or shot.get("prompt", "")
    parts = [self._resolve_element_refs(raw, episode_id)]

    # 1. 导演运镜指令（已有）
    director_text = self._build_director_visual_action_text(shot.get("visual_action"))
    if director_text:
        parts.append(f"导演运镜要求：{director_text}")

    # 2. 新增：基于结构化字段的运镜描述
    elif shot.get("camera_movement"):
        movement_desc = CAMERA_MOVEMENTS.get(shot["camera_movement"], {}).get("desc", "")
        parts.append(f"运镜：{movement_desc}")

    # 3. 新增：时长规则
    duration = shot.get("duration", 6)
    parts.append(f"镜头时长严格为 {duration} 秒")

    # 4. 新增：情绪强度引导
    emotion = shot.get("emotion")
    intensity = shot.get("emotion_intensity", 0)
    if emotion:
        parts.append(f"情绪氛围：{emotion}，强度{'极强' if intensity >= 3 else '强' if intensity >= 2 else '中' if intensity >= 1 else '平缓'}")

    # 5. 新增：禁止文字规则
    parts.append("画面中禁止出现任何字幕、水印、文字标识")

    # 6. 数字人约束（已有）
    dh = self._build_digital_human_constraints_for_episode(...)
    if dh:
        parts.append(dh)

    return "，".join(parts)
```

## 五、关键设计决策

### 5.1 新增字段 vs visual_action 共存策略

**决策**：两者共存，互不冲突。

- `shot_size` / `camera_angle` / `camera_movement` / `emotion` / `emotion_intensity` → LLM 规划自动填充的**扁平字段**，用于快速查询和展示
- `visual_action` (JSON) → 用户通过 3x3 设计器**手动精调**的结构化数据

**同步规则**：
- LLM 规划输出时，同时填充扁平字段和 visual_action 基础值
- 用户在 3x3 设计器中手动调整后，visual_action 的值优先
- 视频生成时：如果 visual_action 有内容，使用 `_build_director_visual_action_text`；否则 fallback 到扁平字段构建简化运镜描述

### 5.2 关键帧的定位

**决策**：关键帧作为**可选增强**，不强制要求。

- 默认工作流：首帧 → 视频生成（现有流程不变）
- 增强工作流：首帧 → 关键帧 → 尾帧 → 视频生成（first_last 模式 + 关键帧作为视觉参考）
- 关键帧不直接参与视频生成的 API 调用，但作为**提示词上下文**注入视频 prompt（"关键帧画面参考：[关键帧描述]"）

### 5.3 向后兼容

**决策**：所有新增字段均有默认值，不影响现有项目。

- 新字段全部 DEFAULT '' 或 DEFAULT 0
- `_build_shot_image_prompt` 在新字段为空时 fallback 到现有行为
- episode_planning 提示词改造通过 custom_prompts 机制支持用户回退到旧版

### 5.4 提示词模板层级

```
用户自定义（series 级） > 用户自定义（全局级） > 新增精细化模板 > 现有默认模板
```

新增模板不覆盖用户已有的自定义提示词。

## 七、风险与注意事项

1. **LLM 输出稳定性**：新增字段越多，LLM 遗漏字段的概率越高。需要在 `studio_service.py` 的规划解析逻辑中添加字段默认值 fallback
2. **提示词长度**：注入分镜理论会增加 system_prompt 长度，需监控 token 消耗。建议仅在"精细模式"下注入完整理论
3. **现有项目兼容**：已有的 series/episode 不含新字段，编辑时需要优雅降级
4. **性能**：关键帧生成增加了一次图片生成调用，批量生成时间会增加。应作为可选步骤
5. **VisualActionDesigner 同步**：扁平字段与 visual_action JSON 的双向同步需要仔细处理，避免数据冲突
